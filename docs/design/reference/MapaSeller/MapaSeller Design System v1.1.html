App Shell

Estrutura base de qualquer tela do MapaSeller: sidebar fixa à esquerda, topbar sticky no topo, conteúdo em coluna única com largura máxima. A sidebar usa três grupos rígidos — Operação, Inteligência, Sistema — refletindo a hierarquia mental do produto.

Developer Migration Plan

Ordem rígida de implementação. Cada fase só começa quando a anterior estiver mergeada e testada. Não pule fases — fundação primeiro, depois telas.

1

Fundação

Tokens globais e shell. Sem isso, qualquer tela vai derivar.

  • Tokens globais em :root
  • AppShell (grid sidebar + topbar)
  • Sidebar com 3 grupos
  • Topbar sticky com search
  • Roteamento React Router
  • Layout responsivo base
2

Componentes base

Biblioteca atômica. Tudo daqui em diante reusa.

  • Button (5 variants + icon)
  • Input, Search, Toggle, Checkbox
  • Badge (6 variants + dot)
  • Chip (filter)
  • KPI Card
  • Data Table (head + row + cell)
  • Modal/Confirmation
  • Upload Dropzone
  • Empty / Loading / Error states
  • Skeleton
3

Telas de operação

As 3 telas que o seller usa todo dia.

  • Dashboard (KPIs + chart + alerts + table)
  • Importar XLSX (4 estados)
  • Produtos (table + filtros + bulk)
4

Telas de inteligência & sistema

Diferencial competitivo do produto.

  • Inteligência ML (URL analyzer + result)
  • Central de Anúncios (editor + preview)
  • Oportunidades (ranking + cards)
  • Configurações (7 seções)
5

Refinamento & QA

Estado final antes do release.

  • Responsividade nos 5 breakpoints
  • Estados de loading em todas as rotas
  • Estados de erro tratados
  • Acessibilidade (focus ring, ARIA)
  • Testes visuais (Storybook)
  • Auditoria final contra esta spec

Design Contract

Regras inegociáveis. Qualquer PR que viole uma delas deve ser revisado antes do merge. Não são preferências — são limites do produto.

Não usar visual dark/cyber como base

MapaSeller é light-first. Dark mode é opcional (em beta) — nunca o padrão. Layouts não devem ser projetados pensando em fundo escuro.

Não usar brilho excessivo

Gradiente é reservado para um único elemento por tela (hero de oportunidade, leader card). Glow, neon e fundos vibrantes são proibidos.

Não apertar textos

Padding mínimo em cards é 20×22. Botões 36px de altura. Hit target mínimo 44px no mobile. Tabelas com 14px vertical.

Não criar cards sem respiro

Gap mínimo entre cards é 20px (–ms-grid-gap). Cards colados (gap 0 ou border-shared) só em casos justificados pela hierarquia.

Não usar cores de marketplace dominando

Amarelo ML, vermelho Shopee, preto Amazon: apenas em ícone 24–32px ou logo. Nunca como bg de seção ou cor de CTA.

Não usar emoji como ícone

Iconografia é Lucide outline 1.75px. Emoji não é icon system — quebra acessibilidade, inconsistente entre OS e desalinha visualmente.

Não misturar estilos

Inter + JetBrains Mono. Cantos sempre arredondados (radius da escala). Ícones outline. Não introduzir font, radius ou estilo de ícone novos.

Não quebrar hierarquia numérica

KPI 34/500. Valor de tabela 14 (numérico em mono 13). O número deve ser maior que seu rótulo — sempre.

Não usar animação como decoração

Transition apenas em hover/focus/state changes (120–200ms ease-out). Loading e shimmer são funcionais. Nada de parallax, particles ou efeitos de "vida".

Não alterar funcional sem necessidade

Esta spec dita aparência e estados. Não mude o comportamento de uma rota durante a implementação visual — escolha um e abra ticket separado.
